### ЧАСТЬ 3: ТЕХНОЛОГИИ (1.5 МИНУТЫ)

**Backend:**
"На backend я использовал:
- Python 3.11 - язык программирования
- aiogram 3.28 - фреймворк для работы с Telegram Bot API
- SQLAlchemy 2.0 - ORM для безопасности от SQL-инъекций
- asyncpg - асинхронный драйвер для PostgreSQL

**Ключевое слово здесь - асинхронность.** 

Что это значит? Обычный код может обслуживать только одного пользователя в секунду. Асинхронный код - может обслуживать 1000+ пользователей одновременно. Как? Когда один пользователь ждёт ответа от базы данных, система переключается на другого пользователя.

**База данных:**
PostgreSQL - это надёжная реляционная база данных с ACID гарантиями. Это означает, что если электричество отключится посередине транзакции, данные не потеряются.

Я выбрал PostgreSQL вместо SQLite потому, ч��о SQLite не рассчитана на production приложения."

## ❓ ЧАСТО ЗАДАВАЕМЫЕ ВОПРОСЫ И ОТВЕТЫ

### Q1: Почему вы выбрали именно Python?

**Ответ:**
"Python - лучший выбор для Telegram ботов потому что:
1. Есть отличный фреймворк aiogram, специально для этого
2. Простой и читаемый синтаксис
3. Большое сообщество
4. Легко писать асинхронный код
5. Быстро разрабатывать

С другими языками (Go, Java, Node.js) можно было, но Python - самый удобный для этой задачи."

### Q2: Почему вы выбрали PostgreSQL?

**Ответ:**
"PostgreSQL - надежная реляционная база данных:
1. ACID гарантии - если электричество отключится, данные не потеряются
2. Поддержка сложных запросов
3. Отличная интеграция с Python через asyncpg
4. SQLite не подходит для production (не рассчитана на много одновременных пользователей)
5. MySQL тоже подошла бы, но PostgreSQL надежнее"

### Q3: Как вы справляетесь с большим количеством одновременных пользователей?

**Ответ:**
"Благодаря асинхронному коду (asyncio):
1. Один поток Python может обслуживать 1000+ пользователей
2. Пул соединений с БД (всегда 10 открытых соединений)
3. Когда один пользователь ждет ответа от БД, система переключается на другого
4. Это намного эффективнее, чем создавать отдельный поток для каждого пользователя

Результат: один процесс на одном сервере может обслуживать 500-1000 одновременных пользователей."

### Q4: Как вы защищаетесь от SQL-инъекций?

**Ответ:**
"Я использую параметризованные запросы:

❌ НЕПРАВИЛЬНО (уязвимо):
```sql
query = f"SELECT * FROM users WHERE id = {user_id}"
```

✅ ПРАВИЛЬНО (безопасно):
```sql
query = "SELECT * FROM users WHERE id = $1"
# $1 - подставляется параметр, который автоматически экранируется
```

Все специальные символы SQL автоматически экранируются, поэтому даже если пользователь введет:
```
' OR '1'='1
```
Это будет интерпретировано как строка, а не как SQL команда."

### Q5: Почему вы используете FSM (State Machine) для заказа?

**Ответ:**
"FSM позволяет:
1. Отслеживать на каком шаге находится пользователь
2. Сохранять данные между шагами
3. Позволить пользователю 'вернуться' кнопкой Назад
4. Показывать ошибки валидации и оставаться на том же шаге

Без FSM пришлось бы писать намного больше кода и была бы легче потеря данных."
